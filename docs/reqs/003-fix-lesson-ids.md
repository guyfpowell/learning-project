# Remove Day/Lesson Association — Everywhere

## 1. The problem

Lessons are addressed and displayed using a `day` field throughout the entire codebase. This is **fundamentally wrong**. Lessons are not days. The word "day" belongs only to the streak system (consecutive calendar days of study). A lesson has a position in a curriculum, not a day.

The `day` field was introduced as a positional integer (`1`, `2`, `3`…) representing a lesson's sequential position within a skill path level. That concept is valid — the field name is not — and it must be renamed throughout the entire stack:

- Database schema (`Lesson.day`)
- ORM unique constraint (`@@unique([skillPathId, day])`)
- Prisma-generated client compound key (`skillPathId_day`)
- Seed script
- Every API service and controller that reads or writes `day`
- All test fixtures that use `day:`
- All admin UI pages that display or sort by `day`
- Shared TypeScript types
- Python generation and backfill scripts
- All specification and documentation files

Nothing that references `day` as a lesson field should survive this change.

---

## 2. The correct model

### 2.0 Positional field definitions

Four fields describe a lesson's position. They are distinct — do not conflate them.

| Field | Scope | Range | Meaning |
|---|---|---|---|
| `lessonNumber` | within one SkillPath (track + level) | 1..80 or 1..88; never resets | Contiguous delivery-engine position. Topic boundaries are invisible to this counter: topic 1 has lessonNumber 1–8, topic 2 has 9–16, topic 3 has 17–24, and so on. The delivery engine walks `lessonNumber + 1` to find the next lesson. This is the DB column being renamed from `day`. |
| `lessonIndex` | within one topic | 1..8; resets at every topic boundary | Position within its topic. Drives "Lesson N of M" display. A lesson with `lessonIndex = 1` is the first lesson of its topic, regardless of what `lessonNumber` it has in the level. |
| `topicId` | within one track + level | 1..10 (beginner/intermediate) or 1..11 (advanced/expert) | Curriculum-order position of the topic. Stable as long as topics are not reordered. |
| `totalLessons` | per-topic constant | 8 for all topics in the current dataset | Total lessons in this lesson's topic. Used alongside `lessonIndex` to render "Lesson N of M". |

**Key contrast:** `lessonNumber = 20` in AI for PMs beginner means the 20th lesson in that level overall. Its `lessonIndex` might be `4` (4th lesson of topic 3) and `totalLessons` is `8`. The browse chip shows "Lesson 4 of 8"; the delivery engine increments `lessonNumber` to 21.

**`lessonNumber` is not globally unique.** Every SkillPath has its own `lessonNumber = 1`. The unique DB constraint is `(skillPathId, lessonNumber)` — never use `lessonNumber` alone to identify a lesson.

### 2.1 How lessons are addressed

Every lesson has two equivalent addresses, defined in `persistent-ids-lessons.md §2.2`. Both are already implemented in the data — only the DB column name is wrong.

| Address form | Example | Use |
|---|---|---|
| **4-part hierarchical** `trackId.levelId.topicId.lessonIndex` | `9.1.3.4` | Human-readable ID, admin display, authoring |
| **3-part flat** `trackId.levelId.lessonNumber` | `9.1.28` | Delivery engine (next-lesson walk: `lessonNumber + 1`) |

### 2.2 Field name changes

| Current field | New field | Meaning |
|---|---|---|
| `Lesson.day` | `Lesson.lessonNumber` | Contiguous 1..N position within a skill path level; used by the delivery engine |
| `@@unique([skillPathId, day])` | `@@unique([skillPathId, lessonNumber])` | Uniqueness constraint — same semantics, correct name |
| Prisma compound key `skillPathId_day` | `skillPathId_lessonNumber` | Auto-generated by Prisma from the constraint |

`lessonNumber` is not new — it already exists in `generated-lessons.json` and is referenced conceptually throughout `persistent-ids-lessons.md`. The gap is that the seed mapped `lessonNumber → day` on write, preserving the wrong name in the DB. This ticket closes that gap.

### 2.3 Admin display

Replace all "Day N" labels with the correct address form for the context:

| Location | Before | After |
|---|---|---|
| `/admin/lessons` flat list — position column | `{lesson.day}` | `{lesson.trackId}.{lesson.levelId}.{lesson.topicId}.{lesson.lessonIndex}` (composite address) |
| `/admin/lessons/browse` — lesson chip within topic | `Day {lesson.day}` | `Lesson {lesson.lessonIndex} of {lesson.totalLessons}` |
| `/admin/lessons/browse` — slide-in panel subtitle | `{topicName} · Day {lesson.day}` | `{topicName} · Lesson {lesson.lessonIndex} of {lesson.totalLessons}` |
| `/admin/skills/[id]` — drag-to-reorder row label | `Day {index + 1}` | `Lesson {index + 1}` |
| `/admin/users/[id]` — progress table position column | `{p.lesson.day}` | `{p.lesson.topicName} · Lesson {p.lesson.lessonIndex} of {p.lesson.totalLessons}` |

The composite address `trackId.levelId.topicId.lessonIndex` is already fully populated in the DB (from `persistent-ids-lessons.md` chunk 5). No data migration is needed beyond the column rename.
